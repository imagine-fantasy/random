[3:15 — 4:15] Architecture
Show diagram.
"What you just saw is Phase 1. Here's the full multi-agent system we've designed.
At the top — a pipeline failure triggers the orchestrator. The orchestrator routes by failure type and manages the full agent lifecycle using SmartSDK's SequentialAgent workflow.
First — the log triage agent. That's what you just saw running. Reads the log, structured output, done.
Second — the code reader agent. This is where it gets firm-specific. It clones the repository and reads our custom Terraform modules using semantic search — understanding what the module was supposed to do versus what the error is saying. This is the agent that catches the gap between what was reviewed and what actually runs.
Third — the fix agent. This one runs an agentic loop. It generates a candidate fix, runs a local plan or build, checks the output. If errors remain — it iterates. Fix, validate, fix again — until the build is clean. It doesn't stop at one attempt.
Fourth — the PR agent raises the pull request with the fix applied and flags whether this was a runtime issue or a review gap.
Finally — the notification agent sends a Teams message or email. On-call engineer gets a summary, a PR link, and the fix is already waiting for review."

[4:15 — 4:40] Why this matters
"The Terraform angle is particularly important for us. We use custom modules that don't behave like standard providers. Bugs surface at runtime, not at review. We approve the PR, everything looks correct, the apply fails days later. Nobody currently flags that gap.
The PR agent we've designed will — every fix is tagged with whether it was a runtime issue or a review gap. That's visibility into where the process broke, not just what broke. That's how we stop recurring failures."
